# Quickie

Quickie is an iOS launcher built around a single text input: the user types or taps, the app fuzzy-matches against a list of capabilities, and the user decides what to do with the input. "Input text and decide what to do with it after."

## Language

**Action**:
A single invokable capability shown in the fuzzy list (e.g. Open App, Web Search, Copy Snippet, Run Shortcut, Calculate). There is exactly one type of thing in the index — an Action — and every subsystem (fuzzy finder, indexing, favorites, fallbacks) operates on it. An Action has an execution lifecycle (selected → optionally collecting input → presenting results), so a multi-step capability is still a single Action, not a separate concept.
_Avoid_: Command, Workflow (reserved — see below), Item

**Main action**:
The single default behavior an Action (or result row) performs when tapped — open a static [[Custom Action]] in the browser, open a file, run a Shortcut, copy-and-stage a math result. Every row in the result list is presented by its main action. Running a main action **resolves the query**: the input clears back to [[Home]] — except the two staging outcomes ([[Stage]], copy-and-stage), which *replace* the query with the staged text (a fresh, unresolved query), and **opening a file preview**, which leaves the query intact because previewing is part of the searching process, not its resolution (the [[Search Files context]] filter is likewise untouched). A [[Secondary action]] never touches the query.
_Avoid_: Default, primary (use "main action")

**Secondary action**:
A one-shot action reached from the row's long-press menu — the *same* menu that carries Pin/Unpin, not a competing gesture. Most secondary actions operate on a specific result's [[Result content]], and their eligibility is a pure function of that Result content, **not** its content type. The content-keyed set is `copy` and `share` on any value-bearing row (text, URL, number, or Pile entry) plus `revealInFiles` on a file and `edit` on every **user-authored, editable** record — a [[Snippet]] and a static [[Custom Action]] each add `edit` (open its in-app editor) *on top of* copy/share, being stored, titled records that still carry a value, unlike a bare text value or a value-only URL; a [[Shortcut Action]] and a [[Custom Action]] each expose `edit` **alone** — a Shortcut deeplinks into the Shortcuts app's editor for the shortcut by name (`shortcuts://open-shortcut`) and a Custom Action opens its live-mirroring editor, since each is an editable *reference* with no pre-resolved text or URL to copy or share (a Custom Action's URL only exists once its slots are filled); a [[Pile]] entry additionally exposes **Remove from Pile** (its home for per-row discard, since the main action stages rather than removes). The one exception to the Result-content rule is **Copy action deeplink**, keyed by the row's **id** rather than its content: it copies the row's `quickie://run/<id>` URL and is offered on **every row whose deeplink runs to an effect** — a content-less command row included, which therefore exposes it *alone* alongside Pin/Unpin (that row has an addressable id even though it has no Result content). The one row it is **withheld** from is **Save for later**: its main action silently writes the typed query into the Pile, so its `quickie://run/<id>` does nothing without a query and copying it is pointless — and since that row also offers no Pin ([[Favorite]]), its long-press menu is empty (issue #140). New Snippet keeps the verb — run standalone it opens the Snippet editor, a real effect. It always sits last in the menu, after any content verbs (ADR 0017; issue #120). Multi-step actions that need a target, date, or name (Make-reminder, Make-Quicklink) are deferred to a later breadcrumb-seeding slice. Distinct from Fallback/content-on-the-text actions, which appear as their own rows rather than behind long-press.
_Avoid_: Context action, more actions

**Result list**:
The single, reversed (bottom-anchored, best match nearest the input/thumb) list shown while typing. Composed of: fuzzy name-matches over Actions, type-triggered results injected with boosted rank by Dynamic Providers (e.g. a math result on top), and Fallback Actions that consume the raw input text as content. All appear as ranked rows; nothing essential is hidden behind a gesture.
_Avoid_: Results, suggestions, search results

**Settings**:
The app's top-level preferences page and the **per-action hub**, reached like any capability — by typing to surface a "Settings" command row, not via chrome (the old top-right gear button is gone). It has two sections: an **app-level** section (**Appearance** — Light / Dark / System, defaulting to System, applied app-wide; a **Clipboard prefill** on/off toggle; a **Show Recents** on/off toggle for the Frecency list; footed by the [[Paste permission hint]], which replaced the old footer's redundant toggle descriptions) and a **Providers** section — one navigation row per [[Provider]] that pushes that provider's [[Management page]] (with one umbrella exception: the [[System provider]]'s single row covers Reminders and Events, whose own pages are reached through it — or by typing their names). The top-level rows are navigation only; a provider is enabled/disabled and configured *inside* its own page, not here. Settings no longer holds *only* settings (ADR 0019): each provider's page unifies its settings and its content.
_Avoid_: Preferences sheet, manage screen (the per-provider destination is the [[Management page]])

**Settings command row**:
The single typeable command row every [[Provider]] surfaces, whose main action **deeplinks** to that provider's [[Management page]] (`.openPage(.settings(panel:))`; `panel: nil` opens the top-level [[Settings]] page). For content/capture providers it *replaces the target* of the former management-page command row (typing "custom actions" lands on the Custom Actions page, now settings + content in one); for the previously row-less dynamic injectors (Calculator, File Search) it is a brand-new row — so every provider, including a disabled one, is re-enableable by typing its name. The lone exception is [[Settings]] itself, which is **non-disableable** so the recovery path always exists.
_Avoid_: Provider row, config command (it is the "Settings command row")

**Disabled**:
An [[Action]] or provider kind that has been **reversibly hidden** from every surface — typed [[Result list]] rows, the Frecency "Recent" list, the [[Favorites grid]], the outward [[Bridged Action]] set, and the [[Actions widget]] / [[Action control]] picker and render — while its data and configuration are retained. Distinct from **delete** (which destroys): disable is the reversible off-switch, delete the permanent one, and both coexist for user content. It exists at two levels: **kind** (a provider's Enabled toggle, the first entry of its [[Management page]]'s options section) and **instance** (a single Custom Action / Snippet / Shortcut, toggled in that page's actions section; an Indexed Folder gets the same per-row toggle on the File Search page, hiding its files while the grant is retained). A disabled kind short-circuits its instances, and the [[System provider]] adds one **umbrella** level above kind: its Enabled toggle off short-circuits every member kind beneath it (Reminders, Events, and its own direct actions), while the member kinds' own toggles keep working underneath. Fallback **activation** and instance disable are related but distinct: **demoting** a fallback to the [[Fallback list]]'s disabled pool (its red minus) hides it from the bottom region only, leaving the action itself live and name-matchable; **instance-disabling** the underlying action hides it everywhere **and** demotes it out of the enabled list into the pool — so a disabled action is never active, and re-enabling leaves it in the pool until the user promotes it again (it does not restore its old rank). The action's enable/disable toggle appears both on its home page and on the [[Fallback list]] itself. A [[Pile]] entry is deliberately **not** instance-disableable: a deferred query is acted on or kept waiting in results — "kept but hidden" is not a state that data has, so its only verbs are stage and discard. Disabling a pinned action keeps the pin (restored on re-enable) but drops it from the grid without consuming a visible slot.
_Avoid_: Hidden, off, archived, soft-deleted (it is "disabled")

**Management page**:
Each [[Provider]]'s own full-screen page — its **settings page and content surface unified** into one (ADR 0019; formerly a separate per-action settings panel plus a content page). Reached by typing the provider's [[Settings command row]] or from the [[Settings]] Providers list, and addressable on its own so the command row can **deeplink** to it. Two sections: (1) **Options** — the provider's declared settings (ADR 0020: `toggle`, `choice` static or `dynamic` — the latter fed live options by the app, e.g. the EventKit calendar list — and `stepper`), with the provider-level **Enabled** toggle (the [[Disabled]] *kind* switch) as the first entry; and (2) the provider's **actions** — each row an [[Action]] with its own enable/disable toggle (the [[Disabled]] *instance* switch) plus **swipe-to-delete** for deletable actions (permanent built-ins like Save for later / New Snippet are disable-only). Providers with no enumerable instances (Calculator, File Search) show only the Options section; the capture providers (Reminders, Events) show their [[Capture step plan]] — a reorderable step double-list — beneath Options. Presents full-screen with its own dismiss affordance — never a partial-height sheet; a bespoke sub-view escape hatch is reserved for options no schema case can express. The **Shortcuts** page is the home for the user's imported [[Shortcut Action]]s: its actions section lists each by name with a per-row "accepts input" toggle (the only way Quickie learns a Shortcut takes input, since import is names-only — and what makes it **fallback-eligible**, entering the [[Fallback list]]'s disabled pool) and swipe-to-delete, and its options section hosts the Sync-Shortcut install/re-sync entry point. There is no manual add — the list is populated solely by the Sync Shortcut import. The **Custom Actions** page is where a [[Custom Action]] is created and edited: its options section hosts the **Browse catalog** navigation row (the [[Catalog]]'s single entry point, the Sync-Shortcut precedent), and its actions section lists each with a per-row enable/disable toggle, swipe-to-delete, and tap-to-edit into the **live-mirroring editor** — a form whose [[Argument]] rows *are* the URL template's `{name}` slots (typing the URL grows/shrinks the rows hard, renaming a row rewrites its `{token}`, and dragging the rows sets the **fill order** the breadcrumb asks in, independent of the URL's order), with Save gated on a named, schemed URL carrying **zero or more** slots (a slot-less URL is a valid static link, ADR 0030); there is no fallback control here — a text-first fill order makes the action fallback-eligible automatically (see [[Fallback Action]]).
_Avoid_: Sheet, manager, Settings panel, Manage screen (it is one page: settings + content)

**Highlighted result**:
The single best result row — `results[0]`, rendered nearest the input and thumb — shown with distinct emphasis so it reads as the default. It is always the nearest-thumb row regardless of how it earned its place (a boosted Calculator/Dynamic result, the best name-match, or the top Fallback when nothing else matches). Pressing **Enter** runs exactly this row's main action. Its Enter intent is signalled two ways: the keyboard's Return key maps to the closest system submit label (e.g. `.search` for a web query, `.go` for a link), and the row itself carries a `⏎`-plus-main-action-glyph hint showing precisely what Enter will do. On Home (empty query) there is no highlighted result and Enter does nothing.
_Avoid_: First result (ambiguous: first-in-array vs first-on-screen), default row, selected result

**Workflow**:
Reserved term, not yet built. A future user-composed chain of multiple Actions, where each Action's output content feeds the next Action's input (validated by content type), authored as visual step-chaining rather than a text DSL. Made possible by every Action declaring typed input/output content from day one. Do not use "Workflow" to mean a single multi-step Action.

**Content type**:
The kind of a value flowing through Quickie — text, url, file, number, date, etc. An item's content type determines which Actions are eligible for it (and their ranking), which secondary actions a result exposes, the **input method** used when it is collected as an Argument (e.g. `date` → an in-place date picker), and — in a future Workflow — whether one Action's output can feed another's input.
_Avoid_: Data type, kind, payload type

**Result content**:
The concrete value or reference a result carries — the thing a [[Secondary action]] operates on — distinct from both its **content type** (the *kind*) and its **main action** (what tapping it does). Either an inline value (a Snippet's or Calculator's text, a static Custom Action's URL) or an edge-resolved reference the app dereferences on demand (a [[Pile]] entry's text by id, a File by bookmark + relative path, a [[Shortcut Action]] by name, a Snippet / [[Custom Action]] (static or slotted) by the stored record's id so the **Edit** verb can open its editor). A command or capture row has **no** Result content, which is what makes it ineligible for secondary actions regardless of its content type (ADR 0017). A Shortcut and a Custom Action are the references carrying no textual value — a Shortcut's name and a Custom Action's id are pure handles for the **Edit** verb, so each is eligible for that verb alone (no copy/share); a static Custom Action, though it carries a real URL to copy/share, likewise keys its Edit off its stored id. Declared per Action, not derived from the main-action outcome (a Shortcut's run outcome is empty yet it declares its name; a Calculator reads as `number` though it copies text).
_Avoid_: Payload, value, content (ambiguous with content type)

**Fallback Action**:
An Action **activated** in the [[Fallback list]]'s enabled section: always surfaced in the result list's bottom fallback region, its point being to **consume the user's typed query as its first Argument** rather than to be found by name-matching. Selecting one **seeds-and-commits**: the query text is committed as Argument 1 through the normal breadcrumb engine — a single-Argument fallback (web search, Save for later, New Snippet, a [[Shortcut Action]] that accepts input) therefore completes in one tap, while a multi-Argument one continues the breadcrumb at step 2 with pill 1 sealed (re-editable like any pill). There is **no fallback flag**: an Action is **fallback-eligible** by *shape* — its first Argument is free text (a date- or choice-first Action has nowhere to put the query) — so every text-first [[Custom Action]], every Shortcut with accepts-input on, the two built-in captures, and the two argument-collecting captures ([[Reminder]] and [[Event]], whose first step is always a free-text title, so the query seeds the title and the breadcrumb continues) are eligible automatically. Eligibility only admits an Action to the Fallback list's disabled pool; the user **activates** it there. Being a fallback is orthogonal to what the Action is — an activated Custom Action or Shortcut is still startable verb-first, where the breadcrumb begins empty. The single result list interleaves fallbacks with name matches; the user resolves intent by choosing a row, never by a mode toggle.
_Avoid_: Default action, catch-all, fallback flag (retired — eligibility is derived from shape; activation lives on the [[Fallback list]])

**Quicklink**:
*Retired term (ADR 0030) — folded into [[Custom Action]].* A Quicklink was a stored *static* URL with no `{slot}` that opened directly; it is now simply a **slot-less Custom Action** — the same concept with zero arguments. The kept term still names that shape informally: a "static Custom Action" or "static link" wears the link glyph, opens directly, and carries copy/share **and** Edit. The former separate Quicklinks page, storage, and provider are gone; all links live under Custom Actions.
_Avoid_: Bookmark, URL action (use "static Custom Action" / "static link"); do not describe it as a distinct concept from Custom Action

**Custom Action**:
A user-authored Action defined by a **URL** with **zero or more** `{slot}` tokens (ADR 0021, 0030). **With slots**, they become the breadcrumb's ordered, typed [[Argument]]s and the final commit opens the filled URL (e.g. `things:///add?title={…}&when={…}`); such an action is automatically **fallback-eligible** when its first fill-order Argument is free text (no flag; see [[Fallback Action]] — activation happens on the [[Fallback list]], where a newly eligible action waits in the disabled pool). **With no slot**, it is a *static link* (the former [[Quicklink]]): a resolved URL that opens directly, consumes no typed text, wears the link glyph, and is **not** fallback-eligible. Either way it matches by name/alias like any Action — verb-first selection of a slotted one starts the breadcrumb empty at Argument 1 — and because submission opens a URL it hands off to another app or the browser, so it is **not** a [[Quick capture]]. Web search is a default-seeded one-Argument Custom Action, pre-enabled as a fallback (a normal, fully deletable entry); the former "Fallback query" kind is absorbed wholesale into this concept. Custom Actions are **default-seeded** on first run — the templated web search, App Store search, Wikipedia, YouTube search, and Google Maps (pre-enabled as fallbacks), plus the static site links YouTube, Gmail, and GitHub (not fallbacks) — under fixed, well-known ids (the CloudKit dedup convention); each is an ordinary, fully deletable entry, and a deleted seed never re-seeds itself (the [[Catalog]] is the way back). A Custom Action result also carries an **Edit** [[Secondary action]] opening the same live-mirroring editor the Custom Actions page uses. A slotted one exposes Edit **alone** (its URL only exists once filled, like a [[Shortcut Action]]); a static one adds Edit *on top of* copy/share (it carries a resolved URL — the Snippet pattern, for a URL). Every enabled Custom Action is also exposed outward as a [[Bridged Action]].
_Avoid_: URL Action, Template action, Quick Action (an iOS home-screen term), Fallback query (retired — absorbed here), search action

**Catalog**:
The built-in, read-only gallery of [[Custom Action]] templates ("catalog entries"), grouped by category — search engines, AI chats, reference & site search, app captures, communication & utilities, and **Sites** (static slot-less links — the former Quicklinks, ADR 0030) — reached via a **Browse catalog** navigation row in the Custom Actions [[Management page]]'s options section. **Install** stamps an entry out as an *ordinary* Custom Action — fully editable, deletable, synced, with **no link back** (no update channel, no managed state) — and **always creates a new row under a fresh id**: the Catalog tracks no installed state, every entry always offers Install, and restoring a deleted action is simply installing again (a new identity — no Frecency revival). Only the first-run **default seeds** carry fixed ids, and that is the seed path's doing, not the Catalog's. Entries whose URL scheme needs a specific app carry a "Requires <app>" note; templates ship in English/US defaults and are edited like any Custom Action after install. A mailto/SMS catalog entry hands off to another app, so it is *not* a [[Quick capture]].
_Avoid_: template gallery, store, marketplace, presets; also do not confuse with the informal "catalog" used for the engine's indexed action set ([[File Search]]) or the widgets' eligible-action snapshot (ADR 0027) — this Catalog is the user-facing gallery page

**Fallback list**:
The Fallbacks [[Management page]]'s two-section edit surface — the same shape as editing the app row of the native iOS share sheet. The **enabled section** is the user-ordered list of active [[Fallback Action]]s, read **most-important-first** (the top of the page is the fallback nearest the input/thumb; because the Result list renders reversed, this order is reversed when projected into the bottom fallback region); each row carries a red minus that **demotes** it to the pool. The **disabled pool** below lists every fallback-eligible Action not currently active, alphabetical by title — a waiting room, not a ranking, so it is deliberately not reorderable; each row carries a green plus that **promotes** it to the **bottom** of the enabled section (promotion says "available", not "most important"). Membership in the enabled section is the only fact the user sets here — the pool is derived (everything eligible but not enabled) — and nothing on this page deletes anything: deletion lives on an action's home page. An Action that **loses eligibility** (a Custom Action's first argument stops being free text, a Shortcut's accepts-input turns off, a re-sync drops the shortcut) leaves the page entirely with **no memory** of its rank; regaining eligibility re-enters it as a newcomer in the pool. The page reads like the iOS share sheet's app row: the **enabled section** is always in edit mode, so each active row shows a drag **grip** (reorder) beside its red minus and carries **no** toggle; the action's **enable/disable** toggle lives only on the **pool** rows (the same instance switch its home page shows). Disabling an action (there or on its home page) hides it everywhere **and** demotes it into the pool, so the pool holds both not-yet-activated actions (a green plus, ready to promote) and disabled ones (dimmed); re-enabling leaves it in the pool until promoted (no rank restore), and promoting a disabled one re-enables it. The four default seeds and the two captures ship pre-enabled, most-important-first: web search, Wikipedia, YouTube, Google Maps, Save for later, New Snippet; the two built-in captures are permanent members — demotable, never removable. Replaces the previous single-list page with per-row disable toggles.
_Avoid_: Fallback settings, fallback order screen (it is one page, "Fallbacks")

**Argument**:
A typed or picked value an Action consumes during its lifecycle. An Action declares zero or more. They are collected one slot at a time in the single bottom input field, with the active Action and filled slots shown as a breadcrumb/pill (`[New Reminder] ▸ "buy milk" ▸ …`). Each Argument declares a **content type** that determines its **input method**: the single input region morphs to the right control per step — the keyboard for free `text`, an in-place graphical picker for a `date`, and for a fixed set of options a **fuzzy-find that reuses the matcher and the reversed result list** (type to filter, best match nearest the thumb) — so the user is never typing a value the system could pick. Verb-first selection clears the search query and prompts for the first Argument; noun-first (Fallback) selection **commits** the literal typed text as the first Argument (seed-and-commit — see [[Fallback Action]]).
_Avoid_: Parameter, field, input (ambiguous)

**Input method**:
The input control the single bottom field morphs into for the current Argument, chosen by that Argument's **content type**: the keyboard for `text`, the keyboard in a **numeric variant** (the system number pad) for a `number`, an in-place graphical picker for a `date`, and a fuzzy-find over a fixed option set (reusing the matcher and the reversed result list) for a choice. The number pad is a *variant* of the keyboard method — still the keyboard, only laid out for digits — not a separate method, and like every input method it is derived from the content type, never set by hand. The mechanism that lets one input region serve every Argument type without modes — the user never types a value the system could pick.
_Avoid_: Input mode, control, widget

**Snippet**:
A piece of saved, reusable text whose primary action is **Copy** — canned replies, an address, a template the user pastes repeatedly. Stored in Quickie (SwiftData + CloudKit), searchable as Actions, addable via the Share Extension. Quickie deliberately has no automatic clipboard history (iOS forbids ambient clipboard access).
_Avoid_: Clipboard history, clip, stash

**Pile**:
The collection of raw query texts the user saved to deal with later — "queries I didn't want to process on the spot." Replaces the former Note concept wholesale: there is **no title, no reader, and no compose-editor**. Each **Pile entry** is just a block of text, stored in Quickie (SwiftData + CloudKit) and captured **silently** by the "Save for later" Fallback (no app switch, no confirm step). An entry is an Action in the index, fuzzy-matched over its **body text** (there is no title to match), and its main action **stages** the text: it replaces the input query and re-runs the matcher — the same reinjection move as a [[Shortcut Action]]'s returned output — after which the entry **leaves the Pile** (staging consumes it). On the **Pile page** (reached via a "Pile" command row; aliases "pile", "later", "saved"), tapping an entry **stages** it — the same main action as its result row, popping back to the launcher — and swipe-to-delete discards an entry *without* staging it. The entries are temporary, so their page is **purely content to view and act on** — deliberately *not* the Pile provider's settings page (a [[Management page]] reached from the [[Settings]] Providers list), the one carve-out from ADR 0019's unified shape; per-row removal from the Result list is the eventual first [[Secondary action]] ("Remove from Pile"), deferred with the long-press mechanism. Distinct from a [[Snippet]] (reusable copy-out text, which keeps its title and *copies* rather than *stages*).
_Avoid_: Note, Inbox, notes system, memo; the individual item has no proper noun — say "a Pile entry" or "a saved query"

**Stage**:
The main action of a [[Pile]] entry: replace the input query with the entry's saved text and re-run the matcher, rebuilding the Result list as if the user had typed it — then remove the entry from the Pile (staging consumes it). Mechanically identical to a [[Shortcut Action]]'s output reinjection; the difference is only the trigger. The user is left "typing" the deferred query and decides what to do with it next. The **Calculator** reuses this reinjection in its main action: a math result **copies-and-stages** — it writes the answer to the clipboard *and* stages it back into the input so the user keeps calculating from it (`2+2` → `4` → `4 * 3`). Unlike a Pile stage it carries a literal value (nothing to resolve, nothing to consume) and reads as a **Copy** row — the copy is the headline, the staging the "also".
_Avoid_: Open, restore, load, paste (staging re-runs the matcher; it is not a passive paste)

**Quick capture**:
An Action that creates a record from the bottom input *without leaving Quickie* — the family comprising **Save for later** (Quickie-stored, into the [[Pile]]), **Reminder** and **Event** (written to EventKit). Each is silent by default (no app switch) and collects its fields through the breadcrumb, honoring just-in-time permission (ADR 0012). **Save for later** is the one capture that is **not** [[Standalone-runnable]]: its main action *silently* writes the raw typed query into the Pile, so run verb-first with no query it does nothing. **New Snippet** shares the same Fallback shape but *is* runnable — run standalone it opens the Snippet editor (seeded with the empty text), a real verb-first action — as do Reminder and Event, which open the breadcrumb.
_Avoid_: Quick add, capture (ambiguous)

**Standalone-runnable**:
Whether running an Action **verb-first with no typed query** — from an [[Entry surface]], a [[Favorite]] card, an [[Actions widget]] / [[Action control]] button, or a `quickie://run/<id>` deeplink — does something. Two shapes fail it: a [[Pile]] entry (its run *consumes* it, so a bound button dies after one tap) and **Save for later** (its run *silently* writes the absent query into the Pile — nothing to save, nothing to see; detected by its `.saveToPile` outcome, not its kind). Everything else runs to a real effect — including **New Snippet**, whose run opens the Snippet editor, and the argument-collecting captures (Reminder, Event), which open the breadcrumb. The single shared test the standalone surfaces read from so their "is this worth offering?" answer can never drift: an Action is [[Favorite]]-eligible and [[Actions widget]]/control-eligible **iff** it is standalone-runnable, and a row's **Copy action deeplink** ([[Secondary action]]) is offered only when its own run does something (a Pile entry keeps that verb — its run stages, which *is* an effect — so the deeplink is gated on "not Save for later" specifically, not on the full predicate).
_Avoid_: Runnable (ambiguous — everything "runs"), launchable

**Reminder**:
A quick-capture Action that creates an EventKit reminder from the breadcrumb — always a **title**, then whichever of **due date** (with an alarm when a time is given; a date-only due date sets no alarm), **notes** text, **priority** choice (None / Low / Medium / High → EventKit 0/9/5/1), and target **list** the user has turned on, in the order they arranged them (see [[Capture step plan]]). List on asks each time; off routes to a chosen default list. Lives in the system Reminders store, distinct from a **Pile** entry (Quickie-stored deferred query text). EventKit permission is requested just-in-time when the Action is activated, before data entry (ADR 0012).
_Avoid_: Todo, task, alarm

**Event**:
A quick-capture Action that creates an EventKit calendar event from the breadcrumb — always a **title**, then whichever of **start** (a timed start defaults to a one-hour duration; a date-only start becomes all-day; the step off makes it all-day today), **location** text, **notes** text, and target **calendar** the user has turned on, in the order they arranged them (see [[Capture step plan]]). Calendar on asks each time; off routes to a chosen default calendar. Silent by default; a setting instead opens the pre-filled system event editor for final review. EventKit permission is requested just-in-time on activation (ADR 0012).
_Avoid_: Appointment, meeting, calendar entry

**Capture step plan**:
The user's per-capture arrangement of which steps a Reminder or Event collects and in what order — a reorderable two-section list on the provider page (the same On/pool shape as the [[Fallback list]]): an **On** section (ordered, drag to reorder, a red minus turns a step off) above an **Off** pool (a green plus turns a step on). The **title always leads** and is never in the list. The enabled, ordered steps become the breadcrumb after the title, so the plan is the single persisted fact; the disabled pool is derived from the capture's fixed step universe.
_Avoid_: calling it a set of toggles (order matters, and it's one list, not independent switches)

**Clipboard prefill**:
A launch-time offer to seed the input field with the current clipboard contents. Quickie silently checks only whether the clipboard *has text* (metadata, no system banner) and, if so, shows a tap-to-fill paste chip backed by the iOS Paste control — reading the actual content only on tap, never ambiently. Tapping the chip **does raise the iOS paste-permission alert**: the bare system paste control is exempt from that alert, but only when shown untouched, and Quickie deliberately dresses it in the input bar's Liquid Glass (the bare control's opaque platter clashes with it), trading the exemption for the look. That trade-off is accepted, not a bug to fix; the mitigation is the [[Paste permission hint]] — inform the user, don't fight the system.
_Avoid_: Auto-paste, clipboard read; never describe the chip as prompt-free or banner-free (only the *has-text metadata check* is silent — the tap prompts until the user allows pasting in iOS Settings)

**Paste permission hint**:
The footer line under the [[Settings]] app-level section telling the user how to stop the iOS paste-permission alert for good: "To paste without iOS asking each time, set Paste from Other Apps to Allow in iOS Settings.", followed by a tappable **Open iOS Settings** link that deeplinks to Quickie's page in the Settings app (where the *Paste from Other Apps* row lives — the user still flips it to **Allow** themselves). It replaced the footer's former toggle descriptions, which restated what the self-describing toggles already said. Passive and always present — never a popup, banner, or nag (an information popup would fight a popup annoyance with another popup) — and necessarily **blind**: iOS offers no way to read the *Paste from Other Apps* state, so the hint cannot know whether it is still needed.
_Avoid_: Paste tip popup, permission primer (that is the quick-capture pattern, ADR 0012)

**Share Extension**:
The iOS share-sheet entry point that turns a shared **URL** or **plain text** into a stored Action written to the shared App Group store (ADR 0006, 0022). A URL becomes a static [[Custom Action]]; text becomes a [[Snippet]] (the default) or a [[Pile]] entry, chosen in a small **classification sheet** with quick edit. It only advertises for URL and text — images and files are out of scope, and it does not appear in the share sheet for them. Items it writes surface in the app on the next foreground, when the in-memory index is rebuilt from the store. Distinct from a [[Provider]]: it is a *write* surface (a way in), not a source of Actions in the [[Result list]]. (The epic that introduced it, #14, said "text → Snippet or **Note**"; read that as Snippet or Pile — Note was retired by ADR 0018.)
_Avoid_: Share sheet action, importer, Add-to-Quickie extension (it is the "Share Extension")

**Favorites widget**:
The interactive Home-Screen [[Entry surface]] mirroring the [[Favorites grid]]: one button per pinned [[Favorite]], in pin order. A button runs its Favorite's **main action** with as little Quickie as possible: a [[Snippet]] copies **in-place** (no app launch), a static [[Custom Action]] or [[Shortcut Action]] **hands off directly** to the browser / Shortcuts app (a Shortcut's output reinjection still lands back in Quickie, exactly as an in-app run), and an Action that needs input or in-app UI (a [[Quick capture]], a slotted [[Custom Action]], [[Search Files context]]) opens the app **pre-selected** — verb-first, breadcrumb at Argument 1, as if its Favorite card had been tapped on [[Home]]. A button whose Favorite no longer resolves degrades to the clean focused Home, never an error (ADR 0025).
_Avoid_: Interactive widget (say which), quick actions widget

**Actions widget**:
The second interactive Home-Screen [[Entry surface]]: the same 2×2 grid as the [[Favorites widget]] — glyph-only cells in small, titled cells in medium, at most four — but showing a **user-chosen** list of Actions instead of the pinned four. The list belongs to the placed widget instance itself, chosen in the system Edit-Widget sheet (no in-app page holds or mirrors it, and each placed instance carries its own list) as **four ordered slots**, each a single-pick from the catalog — not one multi-select list — so the picker shows one item at a time, the choice is inherently capped at four, and the order is set by which slot holds each action; eligible is **every enabled Action that is [[Standalone-runnable]]** — every enabled Action except a [[Pile]] entry (staging consumes it, so a button bound to one would die after one tap) or **Save for later** (its silent Pile write does nothing without a query; New Snippet stays eligible, since it opens the editor; issue #140) — and [[Disabled]] already hides an action from every surface. A slot left empty is skipped; the filled ones pack toward the front of the grid. Buttons execute exactly as [[Favorites widget]] buttons — the same three-way split, the same Frecency outbox. An unconfigured instance renders a one-line long-press invitation whose whole surface deep-links to a clean focused [[Home]]; a chosen action that no longer resolves (deleted or [[Disabled]]) drops from the grid, its slot degrading to the dashed empty-cell tap target — never blank, never inert, never an error.
_Avoid_: Custom widget (collides with [[Custom Action]]), Launcher widget, configurable Favorites

**Action control**:
The configurable Control Center control that runs **one user-chosen Action**, picked in the control's configuration from the same eligible set as the [[Actions widget]]. Activating it opens Quickie and runs the action **tap-equivalently** (through the shared inbox door, `quickie://run/<id>`): a [[Snippet]] copies, a static [[Custom Action]] opens the browser, a [[Quick capture]] opens its breadcrumb — each the correct outcome, its [[Frecency]] credited by the app's ordinary run path. Unlike the [[Actions widget]] it does **not** perform the three-way split (in-place copy / direct hand-off / open): a Control Center control body is a single, non-branching template (WidgetKit's `ControlWidgetTemplateBuilder` has no `buildEither`), and a single intent can't span the lanes (one `perform()` has one return type, and `openAppWhenRun` is static), so the always-open tap-equivalent run is the one path that serves every eligible action. The configured control wears its action's own glyph and title in Control Center; unconfigured or stale (deleted or [[Disabled]]) it falls back to the app glyph and opens a clean, focused [[Home]] — never inert, never an error. Coexists with the static Quick Capture control (a separate gallery entry).
_Avoid_: Run-action control, shortcut control, custom control

**Entry surface**:
A system surface outside the app that gets the user into Quickie fast — the Home/Lock Screen deep-link widget, the interactive [[Favorites widget]] and [[Actions widget]], the Control Center controls (Quick Capture and the [[Action control]]), and the Action Button. The open-focused surfaces express "I have something new to type *now*": they always land on a clean, focused [[Home]] — a stale query is cleared and a half-filled breadcrumb abandoned — whereas resuming via the app icon or app switcher preserves state. Distinct from the [[Share Extension]] (a write surface, not a way in for typing).
_Avoid_: Launcher widget, shortcut surface, entry point (ambiguous with code entry points)

**Provider**:
A source that contributes Actions to the result list. Every Action originates from exactly one Provider. Providers are either Indexed or Dynamic.
_Avoid_: Source, Extension (collides with iOS app extensions), plugin

**System provider**:
The umbrella [[Provider]] grouping Quickie's OS-integration actions. Its [[Management page]] leads with a **cascading** Enabled toggle (see [[Disabled]] — off silences every member kind), then two navigation rows pushing the **Reminders** and **Events** pages (their options, [[Capture step plan]], and typed [[Settings command row]]s survive intact — only their top-level [[Settings]] rows fold into this one), then its own actions section with one permanent, disable-only built-in: **Open iOS Settings** (no arguments — opens Quickie's page in the Settings app, the only target iOS exposes; a query-driven "search Settings" was investigated and is infeasible, publicly or privately). Reminders and Events remain full Providers in their own right; System is a grouping, not a merge. **App Store Search** is deliberately *not* a System built-in but a default-seeded [[Custom Action]] (`itms-apps://…/search?…&term={query}`, pre-enabled as a fallback beside web search): its target is a slotted URL, so it fits the Custom Action model, is editable and deletable, and will join the [[Catalog]] (#143) — and it runs through the proven breadcrumb path instead of a System action's routing (issue #144; ADR 0029 Update).
_Avoid_: OS provider, built-ins provider, merged capture page (the sub-pages are linked, not merged)

**Indexed Provider**:
A Provider whose Actions are a known, enumerable set, pre-indexed for fuzzy search and re-indexed only when its underlying data changes (Snippets, Custom Actions, Shortcuts, favorites, built-in commands). Queried from the in-memory index per keystroke.

**Dynamic Provider**:
A Provider that computes Actions on the fly from the current query and is never in the fuzzy index (Calculator, Unit Converter, File Search, Web Search / Fallbacks). Queried live per keystroke (debounced, cancellable, may be async), and decides for itself whether it applies to the query.

**Home**:
The empty-query state shown the instant the app opens: the Clipboard prefill chip (when applicable), a **Favorites grid** pinned at the top of the screen over a progressive-blur band, and a Frecency "Recent" list that scrolls *under* that band. The tap-without-typing fast path. On the first keystroke the Favorites grid disappears and the live Result list takes the full height, still scrolling under the same blurred top band.
_Avoid_: Landing, start screen, default view

**Favorites grid**:
The 2×2 grid of small Favorite cards pinned at the top of Home over a progressive blur. Shows **at most four** Favorites, in pin order; it is the launch-time, tap-without-typing surface. Visible only on Home — it vanishes the moment the user starts typing, ceding the screen to results. Replaces the earlier horizontal Favorites chip row.
_Avoid_: Favorites row, favorites bar

**Favorite**:
An Action the user has manually pinned. Capped at **four** (the Favorites grid is 2×2); a fifth pin is refused until one is unpinned. Favorites appear in the Favorites grid on Home, receive a ranking boost in Results, and are exposed outward as [[Bridged Action]]s. Every Action is pinnable **except one that is not [[Standalone-runnable]]** — a [[Pile]] entry (its main action consumes it via [[Stage]], so a pin would outlive its target on first use, a ghost card invisibly holding a grid slot) and **Save for later** (its silent Pile write does nothing run without a typed query, so a pinned card would do nothing; New Snippet stays pinnable — it opens the editor; issue #140). Those rows offer no Pin item, and reconciliation prunes any such pin an older build allowed.
_Avoid_: Pinned, starred, bookmark (bookmark means a static Custom Action here)

**Bridged Action**:
An [[Action]] exposed *outward* through the App Intents bridge (ADR 0009, 0024) as a value of the single parameterized App Shortcut — the set is **derived**, never hand-curated: the union of the user's [[Favorite]]s and [[Custom Action]]s, minus anything [[Disabled]], refreshed whenever pins or Custom Actions change. Invoking one from Siri / Spotlight / the Action Button is **tap-equivalent**: the app opens and behaves exactly as if the user had tapped that Action's result row (a Favorite runs its main action; a Custom Action starts its breadcrumb at Argument 1). A bridged reference that no longer resolves — unpinned, deleted, or disabled since the system last synced — degrades to opening the app on [[Home]]. Distinct from the four **headline App Shortcuts** (Quick Capture, [[Pile]]'s Save for later, [[Reminder]], [[Event]]), which are static phrase-invoked verbs, not members of this set.
_Avoid_: Exposed action, exported action, dynamic shortcut (Apple's mechanism, not the concept)

**Frecency**:
The frequency × recency ranking signal derived from a user's past Action selections. Drives the auto-suggested list on Home and boosts ranking in Results. Distinct from Favorite (which is manual).
_Avoid_: Recents, history, MRU

**Shortcut Action**:
An Action that runs one of the user's iOS Shortcuts by name, via x-callback-url. Quickie opens `shortcuts://x-callback-url/run-shortcut` with `x-success` pointing back at `quickie://`; on `x-success` Quickie **reinjects the returned output as the new query text** — the matcher re-runs over it and the Result list rebuilds as if the user had typed it, so the user decides what to do with it next. Reinjection is unconditional on success: non-empty output gives the user something to act on, and empty output clears the field back to a clean Home ready for a fresh query. Running the shortcut clears the query like any [[Main action]], but the text is held for the callback: `x-error` flashes a failure toast and **restores** the cleared query for retry; `x-cancel` silently restores it likewise — so from the user's view a failed or cancelled run leaves their text in place. This is the deliberate poor-man's precursor to [[Workflow]] (no content-type validation, no declared next step, just text back into the single input), not the Workflow feature itself. The returned output therefore lives in inbound URL-scheme handling, not in an `ActionOutcome` payload. A Shortcut row also carries an **Edit** [[Secondary action]]: its long-press menu deeplinks into the Shortcuts app's editor for the shortcut by name via `shortcuts://open-shortcut` — the same name-keyed identity the run path uses, so no Apple UUID is needed. Opening the editor is fire-and-forget (no `x-callback-url`, no returned output), and since a runnable shortcut has no text, Edit is its only secondary verb. Registered **only** by bulk import via the [[Sync Shortcut]] — there is no manual one-by-one add path (dropped from #13; see ADR 0007), so a user who has never run the Sync Shortcut has zero Shortcut Actions. iOS forbids enumerating the user's Shortcuts through any API, which is why the companion-shortcut import is the sole on-ramp.
_Avoid_: Shortcut command, automation

**Sync Shortcut**:
A Quickie-provided iOS Shortcut the user installs and runs manually. It calls the Shortcuts `Get My Shortcuts` action to collect the names of all the user's shortcuts and returns them to Quickie via the `quickie://` URL scheme, populating the user's Shortcut Actions. Re-run to re-sync; there is no background/automatic refresh.
_Avoid_: Importer, bridge shortcut

**Indexed Folder**:
A folder the user has explicitly granted Quickie access to (via the document picker), persisted as a security-scoped bookmark. Grants are managed on the **Indexed Folders** [[Management page]] — a typed "Indexed Folders" command row (aliases "folders", "file access") opens a full-screen list with per-row removal and an **Add Folder** button that presents the document picker in folder-selection mode. Because bookmarks are opaque and device-specific, grants are stored **device-locally and never synced** (ADR 0016), and a grant whose bookmark no longer resolves is **pruned** rather than left as a dead row. File search is bounded to the union of Indexed Folders — iOS forbids whole-filesystem or global indexing. Filenames within them are indexed for fuzzy matching; results open via QuickLook / share / open-in-place.
_Avoid_: Search scope, watched folder, library

**File Search**:
The **Dynamic Provider** that fuzzy-matches filenames within the union of Indexed Folders. Unlike an Indexed Provider — whose whole catalog the SearchEngine matches and ranks centrally — File Search **owns its own filename index and prefilters it itself**, so a large file set never floods the central catalog, Home, or Frecency. Its handful of survivors are then **scored and placed in the ranked region by match quality** — *not* auto-floated to the top like the Calculator, so an exact command name still outranks a fuzzy filename hit. This makes File Search a **ranked-dynamic** provider (competes on score) as opposed to the Calculator's **boosted-dynamic** behaviour (type-triggered, floats top). Surfaces two ways: **inline-capped** (up to ~3 rows, only on strong matches, while typing a normal query) and the **Search Files context** (uncapped; see below). A file result carries only its Indexed Folder's bookmark identity plus the file's relative path; the app resolves that to a security-scoped URL and opens it — Core never touches the filesystem.
_Avoid_: Spotlight, file provider, file index (that is the data, not the Provider)

**Search Files context**:
The dedicated, uncapped file-browsing surface, entered by **selecting a "Search Files" Action row** — never a chrome mode toggle. Once selected, the input scopes to the file index alone (shown as a breadcrumb, `[Search Files] ▸ …`), full-height, and every keystroke filters only filenames. This keeps faith with the Result list's "intent is resolved by choosing a row, never by a mode toggle": the "dedicated mode" is a selected-Action context reusing the breadcrumb model, not a UI mode. Distinct from the ongoing input filter it maintains, it never *commits* a value the way an Argument does — it is a live scoped filter, not an Argument slot.
_Avoid_: File mode, search mode, files tab
