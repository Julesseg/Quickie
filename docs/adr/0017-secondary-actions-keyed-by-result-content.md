# Secondary actions are one-shot, keyed by a result's content, decoupled from the Workflow substrate

ADR 0008 and 0011 framed the long-press **secondary actions** as a *content-type â†’ applicable-actions registry* that would double as the substrate for a future **Workflow** (one Action's typed output feeding the next). Building issue #19 we deliberately narrowed that: secondary actions are a **self-contained, one-shot** affordance, and their eligibility is a pure function of a result's **`ResultContent`** â€” not a `[ContentType: â€¦]` table derived from `Action.inputTypes`. The Workflow substrate stays where ADR 0011 already put it (`inputTypes`/`outputType`); Workflow will rederive its own "what can consume this type" later rather than reusing this registry.

## Considered options

- **Derive eligibility from `Action.inputTypes`** (a result's secondary actions = catalog Actions consuming its output type). This is the literal ADR-0011 substrate and makes "Make Reminder on text" fall out for free â€” but it forces terminal verbs like Copy/Share/Reveal (which produce no output and cannot chain) to masquerade as Actions, and it drags every multi-step action (which needs a target, a date, or a name) into a "one-shot" menu.
- **A `[ContentType: [SecondaryActionKind]]` table.** Rejected once we decided command/capture rows carry a content *type* (`.text`) but must expose **no** secondary actions: a type-keyed table cannot distinguish a text-bearing Snippet (`.text("body")`) from a text-typed Settings command (no content), because both are `ContentType.text`. Only `ResultContent` â€” which encodes *presence and value*, not just type â€” can express that. (A Shortcut sharpens the point further: it is `text`-typed too, yet its eligible verb set is *neither* the command's empty set *nor* the Snippet's copy/share/edit â€” it is `edit` alone. A type-keyed table has one row for `.text`; the value-keyed switch gives each its own arm.)
- **Chosen: a pure `secondaryActions(for: ResultContent) -> [SecondaryActionKind]` switch.** `.none â†’ []` excludes command rows for free; `.file` adds `revealInFiles`; everything content-bearing gets the universal `copy`/`share`. The "registry" is a switch, not a table.

## Consequences

- **`ResultContent` is a first-class, *declared* property of an `Action`, distinct from its main-action outcome.** A Note's main action is "open reader" but its content is `.noteBody(id:)`; a Snippet's is `.text(body)`. Content is declared per factory (defaulting to a derive-from-outcome helper), because the outcome alone is ambiguous â€” a Calculator reads as `.number` though it copies text, and a Shortcut's run outcome derives `.none` yet it declares `.shortcut(name:)` so it can carry the name its Edit deeplink needs.
- **Core decides eligibility; the App executes.** Edge-resolved content (a Note body in SwiftData, a file behind a security-scoped bookmark) means even Copy cannot run in Core. So `SecondaryActionKind` is a bare verb enum and the App owns every execution and resolution â€” the same defer-to-the-edge pattern as `openNote`/`openFile`.
- **This registry is *not* the Workflow substrate.** Chosen for a smaller, shippable long-press menu; the cost is that Workflow will build its own `inputTypes`-derived eligibility later. Multi-step per-type actions (Make-Reminder, Append-to-Note, Convert, Make-Quicklink) are deferred to that later, breadcrumb-seeding work â€” the `secondaryActions(for:)` switch is where they will slot in.
- **Per-content verbs slot in as new `ResultContent` cases, not a widened `.text`.** The first addition beyond the universal set is **Edit** on a Snippet: because `.text` cannot tell a stored, editable Snippet from a value-only Calculator/Fallback result, a Snippet now declares its own `.snippet(id:)` content (mirroring `.pileEntry(id:)`), and `secondaryActions(for: .snippet)` adds `.edit` on top of copy/share. The App resolves the id to the stored record and opens the same Snippet editor the library uses â€” Core only declares the verb eligible, exactly as with `revealInFiles`. This confirms the switch's shape: a new eligibility is a new content case plus one arm, never a type-keyed table entry.
- **A verb can be *reused* across content cases with a different edge-resolution â€” and a content case can offer a verb *without* the universal copy/share.** Extending Edit to a **Shortcut** proves both: a Shortcut declares `.shortcut(name:)` (mirroring `.snippet(id:)`), and `secondaryActions(for: .shortcut)` returns `[.edit]` **alone** â€” no copy/share, because a runnable shortcut is a launchable reference with no textual value. The App resolves the same `.edit` verb differently per content: a `.snippet` opens the in-app editor, a `.shortcut` deeplinks into the Shortcuts app via `shortcuts://open-shortcut?name=` (name-keyed like the run path, so no Apple UUID is needed and the open is a fire-and-forget hand-off with no `x-callback-url`). So `SecondaryActionKind` stays a small shared vocabulary of verbs while `ResultContent` owns *which* verbs and *how* they resolve.
